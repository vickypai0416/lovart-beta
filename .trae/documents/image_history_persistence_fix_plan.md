# 图片历史中图片仍无法显示 - 修复计划

## 一、问题理解与研究结论

当前现象：
- 图片生成后，切换页面后图片历史仍有记录，但图片不显示。
- 页面无明显报错日志。

已知相关模块：
- 历史记录存取与序列化：`src/lib/history-manager.ts`
- IndexedDB 图片持久化：`src/lib/idb-storage.ts`
- 图片生成器历史读取：`src/components/workflows/ImageGeneratorWorkflow.tsx`

初步判断：
1. 历史数据结构与真实存储状态可能不一致（localStorage 有记录，但 IDB 无对应 blob）。
2. 异步写入 IDB 与立即读取历史存在竞态（保存后立刻切页，IDB 还未写完）。
3. `getImageUrl` 仅按 `imageId` 取 blob，若历史记录被替换/裁剪/冲突，可能回退到空或不可用 URL。
4. 当 URL 为 data URL 或远端临时 URL 时，回退策略不足，导致二次加载失败。

## 二、修改目标与范围

目标：
- 保证图片历史在切页、刷新、重开页面后稳定显示。
- 历史记录中的每条图片都可恢复，不依赖短时有效 URL。

范围（拟修改文件）：
1. `src/lib/history-manager.ts`
2. `src/lib/idb-storage.ts`
3. `src/components/workflows/ImageGeneratorWorkflow.tsx`

## 三、实施步骤

### 步骤 1：补强历史写入一致性
- 在 `saveImageToHistory` / `saveChatImageToHistory` 中，确保“写入历史元数据”和“写入 IDB blob”具备一致性策略：
  - 生成唯一且稳定的 `imageId`；
  - 先写 IDB（或至少可追踪成功/失败），再写入可见历史；
  - 对失败路径写入可观测字段（如 `storageStatus`），避免静默失败。

### 步骤 2：优化 URL 回退链路
- 在 `getImageUrl(imageId, fallbackUrl)` 中增强回退逻辑：
  1) 先读 IDB blob；
  2) 若无 blob 且 fallback 为 data URL，直接返回 data URL；
  3) 若 fallback 为 http(s)，增加可用性检测，不可用时尝试代理转换；
  4) 最终返回明确占位值，避免 `<img src="">`。

### 步骤 3：消除保存后立即读取竞态
- 在图片生成器保存历史后，等待持久化完成再刷新历史列表；
- 对批量图片保存使用 `Promise.all`，确保所有图片保存完成后再 `setImageHistory`。

### 步骤 4：渲染层兜底
- 在 `ImageGeneratorWorkflow` 的历史渲染中：
  - 避免传入空 `src`；
  - 对失效图片显示“重试加载”按钮（触发重新获取 URL）；
  - 增加最小化错误日志（仅开发环境）。

### 步骤 5：数据修复（兼容旧历史）
- 为旧数据提供一次性迁移：
  - 扫描历史项，对 `url` 为空但 IDB 仍有 blob 的项恢复 URL；
  - 对确实不可恢复项标记失效，不阻断其他项显示。

## 四、验证方案

1. 功能验证
- 连续生成 1~4 张图片，立即切页，再返回历史页，图片可见。
- 刷新浏览器后历史图片仍可见。
- 关闭浏览器重新打开后历史图片可见。

2. 稳定性验证
- 快速连续生成多批次图片，历史不出现整批失效。
- 删除单张、清空历史后不影响剩余记录显示。

3. 构建验证
- 执行项目构建，确保无编译错误。

## 五、风险与应对

风险 1：IDB 容量或权限问题导致写入失败
- 应对：捕获并标记失败状态；不写入空 URL 到可见历史；提示用户可重试。

风险 2：老数据格式不一致
- 应对：增加迁移与容错读取，避免一次性清空。

风险 3：代理拉取图片跨域失败
- 应对：优先使用已持久化 blob；代理失败只影响单条并显示可重试。

## 六、预期影响

- 图片历史显示稳定性显著提升。
- 用户切页/刷新后不再出现“历史有记录但图片消失”。
- 对现有功能（生成、下载、删除历史）保持兼容。
