# 用户行为和生图数据收集系统 - PRD

## Overview
- **Summary**: 建立用户行为追踪和生图数据收集系统，用于分析用户使用习惯和生成图片质量，为后续工作流优化提供数据支撑
- **Purpose**: 通过收集和分析用户在应用中的操作行为和图片生成数据，深入了解用户需求和痛点，指导产品优化决策
- **Target Users**: 产品团队、开发团队、运营团队

## Goals
- 收集用户工作流选择数据（聊天助手、图片生成器、电商套图）
- 收集图片生成相关数据（提示词、尺寸、质量、模型选择）
- 收集用户交互行为（按钮点击、页面停留时间、操作路径）
- 收集图片质量反馈（用户评分、使用率）
- 提供数据可视化仪表盘
- 确保数据隐私和合规

## Non-Goals (Out of Scope)
- 不收集用户个人身份信息（如姓名、手机号）
- 不收集敏感数据（如API密钥、完整聊天内容）
- 不进行实时数据推送（离线分析为主）

## Background & Context
- 应用已部署为网页应用，需要了解用户实际使用情况
- 目前缺乏用户行为数据，难以进行数据驱动的优化
- 需要一个轻量级的数据收集方案，不影响用户体验

## Functional Requirements
- **FR-1**: 自动追踪用户工作流切换事件
- **FR-2**: 记录图片生成请求参数（提示词、尺寸、质量、模型、数量）
- **FR-3**: 记录生成结果（成功/失败、耗时、图片URL）
- **FR-4**: 收集用户反馈（图片评分、使用反馈）
- **FR-5**: 提供数据查询API
- **FR-6**: 提供数据可视化仪表盘

## Non-Functional Requirements
- **NFR-1**: 数据收集不影响页面性能（<100ms延迟）
- **NFR-2**: 数据存储安全合规（GDPR/个人信息保护法）
- **NFR-3**: 支持数据导出（CSV/JSON格式）
- **NFR-4**: 仪表盘响应时间 < 3秒

## Constraints
- **Technical**: 使用现有的Next.js框架，可扩展使用Prisma+PostgreSQL或SQLite
- **Business**: 需要确保用户知情同意数据收集
- **Dependencies**: 需要后端数据库支持

## Assumptions
- 用户同意数据收集（通过隐私政策告知）
- 应用已有后端API基础结构
- 可以使用第三方分析工具（如Google Analytics）或自建

## Acceptance Criteria

### AC-1: 工作流切换追踪
- **Given**: 用户在聊天助手、图片生成器、电商套图之间切换
- **When**: 用户点击工作流切换按钮
- **Then**: 系统记录切换事件（时间戳、源工作流、目标工作流、用户会话ID）
- **Verification**: `programmatic` - 检查数据库中是否有对应的事件记录

### AC-2: 图片生成参数记录
- **Given**: 用户提交图片生成请求
- **When**: API接收到生成请求
- **Then**: 系统记录完整的请求参数（提示词、尺寸、质量、模型、数量、时间戳、用户会话ID）
- **Verification**: `programmatic` - 检查数据库中是否有完整的生成记录

### AC-3: 生成结果记录
- **Given**: 图片生成完成（成功或失败）
- **When**: API返回结果
- **Then**: 系统记录结果状态、耗时、图片URL（成功时）、错误信息（失败时）
- **Verification**: `programmatic` - 检查数据库中是否有结果记录

### AC-4: 用户反馈收集
- **Given**: 用户对生成的图片进行评分或反馈
- **When**: 用户点击评分按钮或提交反馈表单
- **Then**: 系统记录评分（1-5星）、反馈内容、对应图片ID
- **Verification**: `human-judgment` - 检查反馈是否正确显示在仪表盘

### AC-5: 数据仪表盘
- **Given**: 管理员访问数据仪表盘页面
- **When**: 页面加载完成
- **Then**: 显示关键指标（总生成次数、成功率、平均耗时、热门功能、用户留存）
- **Verification**: `human-judgment` - 检查仪表盘是否显示完整数据
